# UVA10252 優化對比分析

## 檔案對照

- 原始版: UVA10252-easy.py
- 優化版: UVA10252-optimized.py

## 問題核心

目標是在 Manhattan 距離下找最小總距離與最優點數量。

- 最小距離點在 x、y 各自的中位數（偶數時為中位數區間）
- 解的個數 = x 可選數量 * y 可選數量

兩版的數學正確性一致。

## 原始版缺點與優化點

### 1) 距離計算用兩次逐點 abs 累加，常數較大

原始版：
- `sum(abs(x - mx) for x, _ in pts)`
- `sum(abs(y - my) for _, y in pts)`

缺點：
- 每筆測資做兩次 Python 層的 `abs` 迴圈，對大 n 常數成本明顯。

優化版：
- 先排序後建立 prefix sum。
- 用公式在 O(1) 求單軸距離：
  - 左半: $m \cdot k - \text{sumLeft}$
  - 右半: $\text{sumRight} - m \cdot (n-k)$
- 透過 `bisect_right` 取得 $k$。

效果：
- 避免逐點 `abs`，常數更低，對大資料更有感。

### 2) 使用 `pts = [(x, y), ...]` 造成額外 tuple 配置與解包

原始版：
- 每點建立 tuple，再在多處以 `(x, _)`、`(_, y)` 解包。

缺點：
- 造成多餘物件與解包開銷。

優化版：
- 直接建立 `xs`、`ys` 兩個陣列，避免中間 tuple 容器。

效果：
- 記憶體較省、迴圈更直接。

### 3) 邏輯分散於點對容器，資料流較不連續

原始版：
- 先從 `pts` 抽 `xs`/`ys`，再回頭掃 `pts` 算距離。

缺點：
- 資料路徑較長，CPU cache 命中不如單軸連續陣列。

優化版：
- 單軸資料一次讀入、排序、前綴和、直接計算。

效果：
- 迴圈更緊湊，常數下降。

## 複雜度對比

兩版同為：
- 時間: $O(n \log n)$（排序主導）
- 空間: $O(n)$

差異在常數：
- 原始版有兩次 `abs` 逐點掃描與 tuple 解包。
- 優化版把距離計算改為 prefix-sum 公式，減少 Python 層迴圈成本。

## 優化版可能缺點

1. 程式可讀性略降：多了 prefix 與二分計算。
2. 小資料下差異不大：當 n 很小，體感加速有限。
3. 仍受排序支配：若要再快，除非用選擇演算法找中位數（但實作複雜且 Python 下未必更快）。
